1. APB Control Bus

APB는 SoC 안의 peripheral register를 단순하고 예측 가능한 방식으로 읽고 쓰기 위한 control bus입니다. 숭실대 인턴 초반에는 APB의 setup-enable 2단계 전송, register map, address decode, read/write timing을 먼저 학습했고, 이 내용은 최종 AXI-to-APB Bridge 프로젝트의 APB slave 구현으로 이어졌습니다.

이 페이지의 범위APB가 왜 control bus로 쓰이는지, setup/enable phase가 어떻게 동작하는지, Lab에서 register block으로 무엇을 구현했는지 정리합니다.

1.1 APB는 어떤 역할을 하는가

SoC를 처음 볼 때 bus를 “데이터가 지나가는 선” 정도로 생각하기 쉽습니다. 하지만 실제 SoC에는 성격이 다른 통신이 섞여 있습니다. 예를 들어 image buffer처럼 큰 데이터를 빠르게 옮겨야 하는 경로도 있고, sensor mode나 interrupt mask처럼 register 몇 개만 설정하면 되는 경로도 있습니다. APB는 후자, 즉 peripheral 설정과 상태 확인을 위한 단순한 register access에 맞춰진 bus입니다.

APB에서 중요한 질문은 “얼마나 많은 데이터를 빠르게 옮기는가”가 아니라 “어떤 주소의 register를 언제 읽고 쓰는가”입니다. software가 특정 주소에 값을 쓰면 hardware 내부 register가 바뀌고, read 요청을 보내면 해당 register 값이 PRDATA로 돌아옵니다. 그래서 APB는 timer, GPIO, sensor control register, configuration register처럼 복잡한 burst가 필요 없는 블록에 잘 맞습니다.

초심자 관점

APB는 주변장치의 설정 버튼을 누르는 통로에 가깝습니다

카메라 센서의 exposure mode를 바꾸거나, 어떤 interrupt를 켤지 정하거나, 하드웨어 블록의 status bit를 읽는 작업은 대용량 데이터 전송이 아니라 register 제어입니다. APB는 이런 제어 작업을 단순한 timing으로 처리하도록 만든 bus입니다.

1.2 Setup phase와 Enable phase

APB transfer는 두 단계로 이해하면 쉽습니다. 첫 번째 cycle의 setup phase에서는 PSEL을 올리고, 주소(PADDR), write 여부(PWRITE), write data(PWDATA)를 준비합니다. 두 번째 cycle의 enable phase에서는 PENABLE을 올려 “이제 실제 전송을 하자”는 상태가 됩니다. 이때 slave가 PREADY를 올리면 transfer가 완료됩니다.

write transaction이라면 enable phase에서 조건이 만족될 때 register가 갱신됩니다. read transaction이라면 선택된 register 값이 PRDATA로 나옵니다. PREADY가 늦게 올라오면 transfer는 기다립니다. 이 ready 지연은 peripheral이 바로 응답할 수 없을 때를 표현할 수 있고, testbench에서는 이 조건을 파형으로 확인할 수 있습니다.

Setup phase
PSEL을 올리고 주소, write/read 여부, write data를 안정적으로 준비하는 구간입니다.
Enable phase
PENABLE이 올라간 상태에서 PREADY를 확인하고, 실제 register access가 완료되는 구간입니다.
Transfer complete
일반적으로 PSEL, PENABLE, PREADY가 함께 만족되는 시점에 write commit 또는 read data valid를 판단합니다.

1.3 Lab에서 구현한 APB register block

APB 실습에서는 주소 decode에 따라 내부 register에 값을 쓰거나, read data mux를 통해 선택된 값을 반환하는 register interface block을 만들었습니다. 처음에는 단순 read/write register를 다루고, 이후 SRAM 접근, endian 변환, interrupt/mask 확장형 APB block으로 이어졌습니다. 이 흐름을 통해 register map이 실제 hardware 기능과 어떻게 연결되는지 익혔습니다.

이 단계에서 익힌 write enable 조건, read enable 조건, PREADY 생성 방식은 최종 AXI-to-APB Bridge 프로젝트에서 APB slave를 구현할 때 그대로 기반이 되었습니다. 즉 APB는 최종 프로젝트의 “목적지 bus”였고, AXI 요청이 마지막에 도착해 실제 register를 읽고 쓰는 쪽이었습니다.

1.4 참고한 공식 자료

AMBA는 Arm이 제공하는 SoC interface/protocol specification 묶음입니다. Arm의 AMBA specification 페이지에는 APB, AHB, AXI가 함께 정리되어 있으며, 이 페이지의 설명은 숭실대 수업자료와 Arm AMBA specification 구조를 기준으로 다시 풀어 썼습니다.
