1. APB Control Bus

APB를 처음 배울 때 내가 잡은 기준은 간단했다. SoC 안에서 모든 데이터가 고속으로 움직일 필요는 없고, 센서 설정값, 상태 register, interrupt mask처럼 “정해진 주소를 읽고 쓰는 제어”가 훨씬 자주 필요하다. 숭실대 인턴의 APB 실습은 이 제어 경로를 Verilog로 직접 만드는 과정이었다. 주소가 들어왔을 때 어떤 register를 고를지, write는 어느 cycle에 commit할지, read data는 어떤 mux를 통해 돌려줄지를 파형으로 확인했다.

이 페이지의 범위 APB setup/enable timing, Lab2 register block, Lab3 SRAM memory-mapped access, Lab4 interrupt register 확장, SimVision 파형 검증을 정리합니다.

1.1 APB를 먼저 배운 이유

수업자료에서는 APB를 느린 주변장치를 제어하기 위한 단순한 interface로 설명했다. 여기서 중요한 말은 “느리다”가 아니라 “제어”였다. 카메라 센서나 peripheral IP를 생각하면, 실제 영상 데이터는 별도 경로로 흐르더라도 exposure, mode, interrupt enable 같은 설정은 register access로 제어된다. APB는 이런 register access를 복잡한 burst 없이 setup phase와 enable phase 두 단계로 처리한다.

Lab2에서는 이 개념을 바로 register block으로 옮겼다. 강의자료의 register map은 입력 A를 0x0000, 입력 B를 0x0004, 출력 C를 0x2000 주소로 두고, A+B 결과가 1 clock 뒤 출력 register에 mapping되는 구조를 제시했다. 내가 확인한 핵심은 주소 숫자 자체보다 주소 decode가 하드웨어 동작으로 바뀌는 과정이었다. PADDR가 특정 주소를 가리키고, PWRITE와 PENABLE 조건이 맞을 때만 내부 register가 갱신된다.

Control

주변장치 설정 경로

대용량 전송보다 register 단위 설정과 상태 확인에 맞는 bus로 이해했다.

Decode

주소가 register 선택으로 바뀜

PADDR 조건을 보고 입력 A/B, 출력 C, control/status register를 선택한다.

Verify

파형으로 cycle 위치 확인

write commit과 read data 반환이 setup이 아니라 enable phase에서 일어나는지 확인했다.

1.2 Setup phase와 Enable phase

APB transfer는 두 박자로 읽으면 이해가 쉽다. 첫 cycle에서는 PSEL을 올리고 주소, write 여부, write data를 안정시킨다. 이때는 아직 transfer가 끝난 것이 아니다. 다음 cycle에서 PENABLE이 올라가고, slave가 PREADY를 올리면 그때 write가 register에 반영되거나 read data가 유효해진다. 그래서 APB RTL을 볼 때는 “PSEL이 올라갔는가”보다 “PSEL, PENABLE, PREADY가 함께 transfer 완료 조건을 만들었는가”를 확인해야 했다.

1Setup

PSEL=1, PENABLE=0. 주소와 방향, write data를 준비한다.

2Enable

PSEL=1, PENABLE=1. slave가 실제 transfer를 처리한다.

3Ready

PREADY=1이면 transfer가 완료된다. wait가 필요하면 enable phase가 유지된다.

4Commit/Return

write는 register update, read는 PRDATA 반환으로 끝난다.

1.3 APB 계열 Lab에서 한 일

Lab2의 register block은 가장 작은 출발점이었다. APB interface block에서 write enable, write data, read enable, read data를 만들고, function block에서는 두 입력을 더해 출력 register에 연결했다. 여기서 APB는 단순히 외부 신호 묶음이 아니라, software address map을 실제 hardware register로 연결하는 약속이라는 감각이 생겼다.

Lab3-1에서는 같은 APB 접근을 4-depth, 32-bit SP-SRAM으로 확장했다. 주소 0x4000, 0x4004, 0x4008, 0x400C가 SRAM entry로 대응되고, APB write/read가 곧 SRAM write/read 제어가 된다. Lab3-2에서는 InBuf와 OutBuf를 두고, 0x0000 start command와 0x0004 packet word size로 endian conversion 동작을 시작시키는 구조를 다뤘다. 이 단계부터 APB는 단순 register 읽기/쓰기가 아니라, 하드웨어 연산을 시작시키는 command interface로 보였다.

Lab4에서는 interrupt enable, pending clear, mask register를 붙였다. 하드웨어가 작업을 끝냈을 때 pending bit를 세우고, software가 이를 읽고 clear하는 흐름까지 들어오면서, register map이 단순 데이터 저장소가 아니라 하드웨어와 소프트웨어가 상태를 주고받는 계약이라는 점을 확인했다.

Lab2

APB register interface

입력 A/B register write, 출력 C read, 1-clock delayed add result를 확인했다.

Lab3-1

APB to SP-SRAM

0x4000 계열 주소가 SRAM entry로 decode되고 read/write 제어로 연결되는지 봤다.

Lab3-2

Buffer + endian conversion

start command, packet size, InBuf/OutBuf를 통해 APB가 연산 시작 interface가 되는 구조를 실습했다.

Lab4

Interrupt register

enable, pending, mask register가 시스템 상태 제어로 이어지는 흐름을 확인했다.

1.4 파형으로 확인한 APB 동작

파형에서는 PSEL과 PENABLE이 순서대로 올라가는지, 주소 decode 결과가 내부 write/read enable과 맞는지, PRDATA가 read cycle에 맞춰 나오는지를 봤다. APB가 쉬운 bus라고 해도, 이 순서를 한 cycle 잘못 보면 register가 의도하지 않은 시점에 갱신되거나 read data가 틀어질 수 있다.
