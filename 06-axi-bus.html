3. AXI Channel Bus

AXI는 address, data, response 흐름을 여러 독립 채널로 나누어 높은 처리량과 유연성을 얻는 AMBA bus protocol입니다. APB가 단순 register 제어에 적합하고 AHB가 phase pipeline을 보여 준다면, AXI는 요청과 응답을 채널별 handshake로 분리해 더 복잡한 SoC 내부 연결을 다룹니다. 숭실대 인턴에서는 AXI의 AW/W/B, AR/R 채널 구조와 valid-ready handshake를 정리했고, 최종 프로젝트에서는 이 AXI 요청을 APB register access로 변환하는 bridge를 구현했습니다.

이 페이지의 범위 AXI write/read 채널, valid-ready handshake, burst/ID/outstanding 개념, 최종 프로젝트에서 single transfer로 범위를 제한한 이유를 정리합니다.

3.1 AXI가 채널을 나누는 이유

AXI를 처음 볼 때 가장 헷갈리는 부분은 신호가 한 묶음으로 움직이지 않는다는 점입니다. write transaction도 주소, 데이터, 응답이 각각 다른 채널로 나뉘고, read transaction도 주소 요청과 data 반환이 분리됩니다. 이렇게 나누면 주소를 보내는 시점, data를 보내는 시점, 응답을 받는 시점을 독립적으로 조절할 수 있습니다.

초심자에게는 AXI를 “여러 차선이 있는 도로”처럼 보는 것이 이해하기 쉽습니다. APB가 한 차선에서 순서대로 register를 읽고 쓰는 느낌이라면, AXI는 write 주소 차선, write data 차선, write 응답 차선, read 주소 차선, read data 차선을 따로 둡니다. 각 차선은 자신의 valid-ready 조건이 맞을 때만 한 beat의 정보를 넘깁니다.

AXI 이해의 핵심

전송은 VALID와 READY가 같은 cycle에 함께 올라올 때 성립합니다

VALID는 보내는 쪽이 “지금 정보가 유효하다”고 말하는 신호이고, READY는 받는 쪽이 “지금 받을 수 있다”고 말하는 신호입니다. 두 신호가 동시에 만족될 때 해당 채널의 transfer가 완료됩니다.

3.2 Write path와 Read path

AXI write는 AW, W, B 세 채널로 구성됩니다. AW 채널은 “어디에 쓸 것인가”를 전달하고, W 채널은 “무슨 값을 쓸 것인가”를 전달하며, B 채널은 “write가 어떻게 끝났는가”를 응답합니다. read는 AR, R 두 채널로 구성됩니다. AR 채널은 “어디를 읽을 것인가”를 전달하고, R 채널은 해당 주소의 data와 응답을 반환합니다.

AW channel
write address와 control 정보를 전달합니다. AWVALID/AWREADY가 동시에 만족될 때 주소 요청이 받아들여집니다.
W channel
write data를 전달합니다. burst transfer에서는 WLAST가 마지막 data beat를 표시할 수 있습니다.
B channel
write transaction의 응답을 반환합니다. bridge 구현에서는 write가 APB 쪽에서 끝난 뒤 B response를 돌려주는 흐름을 만들었습니다.
AR channel
read address와 control 정보를 전달합니다. read 요청의 시작점입니다.
R channel
read data와 response를 반환합니다. APB slave에서 읽은 register 값이 이 채널로 AXI master에게 돌아갑니다.

3.3 Burst, ID, outstanding을 어떻게 바라봤는가

AXI에는 burst, ID, outstanding transaction처럼 고성능 시스템을 위한 기능이 있습니다. burst는 연속된 data beat를 한 transaction으로 묶어 전송하는 개념이고, ID는 여러 transaction을 구분하기 위한 표식입니다. outstanding transaction은 이전 요청의 응답을 기다리는 동안 다음 요청을 추가로 보낼 수 있는 구조를 의미합니다. 이런 기능은 AXI가 APB보다 훨씬 복잡한 이유이기도 합니다.

다만 최종 프로젝트에서는 처음부터 모든 AXI 기능을 구현하지 않았습니다. 먼저 single transfer 중심으로 범위를 좁혀, 하나의 AXI write/read 요청이 APB register access로 정확히 변환되는지를 확인했습니다. 버스 설계에서 중요한 것은 기능을 많이 넣는 것보다, 현재 구현 범위에서 handshake와 응답을 끝까지 닫는 것입니다.

3.4 최종 프로젝트로 이어진 구현 관점

AXI 학습의 목표는 신호 이름을 외우는 것이 아니라, 채널별 책임을 분리해서 보는 것이었습니다. write path에서는 AW와 W가 모두 들어온 뒤 APB write를 시작해야 하고, read path에서는 AR 요청을 받은 뒤 APB read 결과를 R channel로 반환해야 합니다. 이 관점은 AXI-to-APB Bridge에서 write FSM과 read FSM을 분리해 구현하는 기준이 되었습니다.
