2. AHB Pipelined Bus

AHB에서 처음 크게 달라진 부분은 “주소를 보는 cycle”과 “data를 처리하는 cycle”이 분리된다는 점이었다. APB는 setup-enable 두 단계를 같은 transfer 안에서 따라가면 됐지만, AHB는 address phase와 data phase가 겹쳐 흐른다. 그래서 파형을 볼 때 지금 보이는 HWDATA나 HRDATA가 어느 HADDR와 짝인지 한 박자 늦춰서 추적해야 했다.

이 페이지의 범위 AHB topology, HSEL/HTRANS/HREADY, address/data phase, Lab5 AHB register block, Lab6 AHB-to-SP-SRAM wait-state 검증을 정리합니다.

2.1 APB 다음에 AHB를 배운 이유

APB만 보면 SoC bus가 register 제어용 통로처럼 느껴질 수 있다. 하지만 SoC 안에는 CPU, DMA, GPU/NPU, memory controller처럼 더 빠른 데이터 이동이 필요한 블록들이 있다. 수업자료에서는 APB, AHB, AXI를 비교하면서 APB는 단순 주변장치 제어, AHB는 high-speed internal data bus, AXI는 더 큰 확장성과 독립 채널 구조로 설명했다. AHB 실습은 그 중간 단계였다. 아직 AXI처럼 다섯 채널로 쪼개지지는 않지만, APB보다 bus system에 가까운 구조를 가진다.

2.2 Master, decoder, mux, slave 구조

AHB는 master가 주소와 제어신호를 내고, decoder가 주소 범위에 맞는 slave를 선택한 뒤, mux가 선택된 slave의 response와 read data를 master로 돌려주는 구조다. Lab5의 작은 register block만 보면 단순해 보이지만, 실제로는 HADDR가 곧바로 register만 고르는 것이 아니라 HSEL을 통해 어떤 slave를 선택할지 먼저 결정한다. 이 구조를 이해해야 여러 peripheral이 같은 bus에 붙을 때 address map이 왜 중요한지 보인다.

Master

요청 생성

HADDR, HWRITE, HTRANS로 어느 주소를 읽거나 쓸지 낸다.

Decoder

slave 선택

주소 범위를 보고 HSEL을 만들어 해당 slave만 반응하도록 한다.

Mux

응답 회수

선택된 slave의 HRDATA, HREADY, HRESP를 master 쪽으로 돌려준다.

2.3 Address phase와 Data phase

AHB RTL에서 가장 조심한 부분은 phase latch였다. Lab5의 interface block은 HREADYin이 1일 때 HSEL, HTRANS, HWRITE, HADDR를 register에 잡아 두고, 다음 data phase에서 그 latched value를 기준으로 register write/read enable을 만든다. 즉 data phase에서 쓰는 주소는 현재 HADDR가 아니라 이전 address phase에서 잡아 둔 주소다.

이 차이 때문에 파형을 볼 때 HADDR와 HWDATA를 같은 세로줄에서만 보면 헷갈린다. address phase에서 0x7000_0000이 들어오고, 다음 cycle의 data phase에서 HWDATA가 들어올 때 register A write가 일어난다. HREADY가 낮으면 이 진행 자체가 멈추기 때문에, HREADY는 단순 완료 신호가 아니라 bus phase를 전진시킬지 결정하는 신호로 이해했다.

2.4 AHB 계열 Lab에서 한 일

Lab5에서는 AHB slave interface를 가진 register block을 만들었다. register map은 0x0000 입력 A, 0x0004 입력 B, 0x0008 출력 C였고, A+B 결과가 1 clock 뒤 출력 register로 연결됐다. APB Lab2와 기능은 비슷하지만, 구현 기준은 달랐다. APB에서는 PSEL/PENABLE 조건을 봤다면, AHB에서는 HSEL이 선택되어 있고 HTRANS가 NonSeq 또는 Seq이며, HREADYin이 1인 phase에서 latch된 주소를 기준으로 동작을 만들었다.

Lab6에서는 AHB access를 SP-SRAM으로 확장했다. 주소 0x7000_4000, 0x7000_4004, 0x7000_4008, 0x7000_400C가 4개의 SRAM entry로 연결됐다. write는 data phase에서 바로 SRAM에 기록할 수 있어 zero-wait로 처리했다. 반면 synchronous SRAM read는 read data가 다음 clock에 나오기 때문에 one-wait가 필요했다. 이 부분에서 bus protocol과 memory macro timing을 맞추는 일이 별도 설계 문제가 된다는 것을 봤다.

Lab5

AHB register access

HSEL/HTRANS/HWRITE/HADDR를 latch하고 data phase에서 register write/read를 수행했다.

Lab6

AHB to SP-SRAM

write는 zero-wait, read는 synchronous SRAM 특성 때문에 one-wait로 처리했다.

phase와 wait-state

2.5 파형으로 확인한 AHB 동작

AHB 파형에서는 신호 이름보다 phase 대응을 우선해서 봤다. HTRANS가 NonSeq로 올라간 주소 phase, 그 다음 data phase의 HWDATA/HRDATA, 그리고 HREADY가 wait-state를 만드는 구간을 함께 봐야 실제 transfer가 보인다.
