3. AXI Channel Bus

AXI는 AHB보다 “신호가 많다” 정도로 이해하면 금방 막힌다. 내가 Lab7에서 잡은 핵심은 하나의 transaction이 여러 채널로 나뉘고, 각 채널이 VALID/READY handshake로 따로 성립한다는 점이었다. write는 주소, data, response가 분리되고, read도 주소와 data 반환이 분리된다. 그래서 AXI 파형은 한 줄로 읽기보다 채널별로 쪼개서 읽어야 했다.

이 페이지의 범위 AXI 5개 channel, VALID/READY handshake, burst와 response, Lab7 AXI slave register block, 최종 bridge로 이어지는 기준을 정리합니다.

3.1 AXI가 어려웠던 이유: 하나의 전송이 여러 채널로 나뉜다

AHB에서는 address phase와 data phase가 겹치지만, AXI에서는 write address(AW), write data(W), write response(B), read address(AR), read data(R)가 서로 다른 채널로 분리된다. 이 구조 때문에 “주소가 들어왔다”와 “data가 들어왔다”와 “응답을 보냈다”가 같은 cycle에 끝나지 않는다. 각 채널은 VALID와 READY가 같은 cycle에 1이 될 때만 handshake가 성립한다.

강의자료에서는 AXI가 AHB보다 복잡한 대신 independent channel, outstanding transaction, ID 기반 out-of-order response 같은 확장성을 갖는다고 설명했다. Lab7에서 이 모든 고급 기능을 전부 구현한 것은 아니지만, 다섯 채널을 나눠 생각하는 습관은 최종 AXI-to-APB bridge를 만들 때 그대로 필요했다.

AWWrite address

어느 주소에 쓸지 전달한다.

WWrite data

실제 data와 마지막 beat 표시를 보낸다.

BWrite response

write가 끝났는지 OKAY/ERROR response를 반환한다.

AR/RRead path

read 주소를 받고 R channel로 data와 response를 돌려준다.

3.2 VALID/READY를 어떻게 봤는가

AXI에서 VALID는 보내는 쪽이 “지금 정보가 유효하다”고 내는 신호이고, READY는 받는 쪽이 “지금 받을 수 있다”고 내는 신호다. 둘 중 하나만 올라가면 아직 transfer가 아니다. 이 규칙이 단순해 보이지만, 채널이 다섯 개라서 파형에서는 AW handshake, W handshake, B handshake, AR handshake, R handshake를 각각 따로 표시하며 봐야 한다.

Lab7에서는 write path에서 AW handshake로 주소를 받고, W handshake로 data를 받은 다음 BVALID/BREADY로 write response를 닫았다. read path에서는 AR handshake 이후 RVALID/RREADY와 RLAST를 확인했다. 이 과정을 거치면서 AXI RTL은 “한 상태에서 모든 일을 처리하는 회로”가 아니라, 채널별 완료 조건을 분리해서 잡아야 하는 구조라는 점을 알게 됐다.

3.3 Lab7에서 한 일: AXI slave register block

Lab7은 AXI slave interface를 가진 register block 실습이었다. register map은 0x7000_0000 입력 A, 0x7000_0004 입력 B, 0x7000_0008 출력 C로 구성되었고, 내부 function block은 A+B 결과를 출력으로 만든다. APB Lab2, AHB Lab5와 같은 add register 예제를 AXI channel 구조로 다시 구현한 셈이다.

이 실습에서 본 핵심은 기능보다 interface였다. 같은 “register A에 값을 쓴다”는 동작도 AXI에서는 AW 채널로 주소를 받고, W 채널로 data를 받은 뒤, B 채널로 OKAY response를 돌려줘야 끝난다. read도 AR 채널에서 주소를 받은 후 R 채널로 data, response, last 정보를 돌려줘야 한다. 그래서 Lab7 파형에서는 add 결과 자체보다 채널별 handshake 순서와 response가 맞는지를 주로 확인했다.

3.4 AXI를 배운 뒤 Bridge로 넘어간 이유

AXI slave register block을 구현한 뒤에는 AXI 요청을 다른 protocol로 바꾸는 문제가 자연스럽게 나온다. 실제 SoC에서는 CPU나 고성능 master가 AXI로 접근하고, UART, I2C, timer 같은 주변장치 register는 APB 쪽에 붙는 경우가 많다. 즉 AXI를 잘 아는 것만으로는 부족하고, AXI 요청을 APB transfer로 변환하는 중간 회로가 필요하다.

그래서 최종 프로젝트에서는 AXI의 AW/W/B, AR/R handshake를 APB의 setup-enable transfer로 바꾸는 AXI-to-APB bridge를 만들었다. Lab7에서 채널별 handshake를 분리해서 본 경험이, write FSM과 read FSM을 따로 설계하는 기준이 됐다.
